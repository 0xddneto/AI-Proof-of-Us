# AIPOU Outreach Sprint - 2026-07-01

This log tracks the long-running outreach sprint requested for AIPOU. The sprint is intentionally safety-first: research, AI-to-AI review, target mapping, and prepared messages are recorded here. No external forum, issue, discussion, DM, or social post should be sent without destination-specific confirmation.

## Sprint Frame

AIPOU should be introduced as:

```text
MCP receipts for AI agents.
```

Not as:

```text
Earn tokens for using AI.
```

The developer-facing pitch:

```text
AIPOU is an MCP-first protocol that lets agents create signed, privacy-preserving receipts for useful AI-assisted work; AIPOU is the Base reward token attached only after receipts are validated.
```

- OpenClaw/ClawHub outreach must lead with supply-chain security because skill security is an active concern.

### LLMOps / Observability

Targets:

- Helicone
- Langfuse
- OpenLLMetry / Traceloop
- PostHog AI Observability
- Awesome LLMOps lists

Why it matters:

- These projects already care about traces, costs, latency, sessions, prompts, evaluations, and agent workflows.
- AIPOU receipts can be positioned beside observability traces, not as a replacement.
- OpenLLMetry has active discussion around MCP instrumentation, hiding trace content, Ollama inference metadata, and agent-to-agent distributed trace context.
- Helicone has active discussion around browser-session correlation for increasingly agentic workflows.
- PostHog AI Observability documents traces, costs, latency, errors, and full sessions as first-class debugging concepts.

Potential angle:

```text
AIPOU receipts could become a portable, signed companion to traces: a minimal proof that a task lifecycle happened, without exposing raw prompts or outputs.
```

Draft GitHub Discussion/Issue comment:

```text
I am exploring AIPOU, an MCP-first receipt protocol for AI agents. It does not replace traces; it creates signed task lifecycle receipts that can reference hashes, model/client metadata, duration, and validation status without storing raw prompts or outputs. Would a signed-receipt layer be useful beside LLMOps traces for audit, provenance, or cross-client agent workflows?
```
