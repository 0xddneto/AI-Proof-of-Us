# AIPOU Outreach Sprint - 2026-07-01

This log tracks the long-running outreach sprint requested for AIPOU. The sprint is intentionally safety-first: research, AI-to-AI review, target mapping, and prepared messages are recorded here. No external forum, issue, discussion, DM, or social post should be sent without destination-specific confirmation.

## Sprint Frame

AIPOU should be introduced as:

```text
MCP receipts for AI agents.
```

Not as:

```text
Earn tokens for using AI.
```

The developer-facing pitch:

```text
AIPOU is an MCP-first protocol that lets agents create signed, privacy-preserving receipts for useful AI-assisted work; AIPOU is the Base reward token attached only after receipts are validated.
```

### MCP Protocol / SDK Discussions

Targets:

- MCP client-experience feedback proposal.
- MCP server cards and discovery proposal.
- MCP session/lifecycle discussions.
- Agents SDK `_meta` response issue.

Why it matters:

- AIPOU receipts are effectively lifecycle metadata: task started, task completed, receipt signed, validation status.
- MCP discussions already care about client experience, metadata propagation, discovery, sessions, and lifecycle cleanup.
- AIPOU should avoid trying to change the core protocol too early; it should ask where receipt metadata belongs.

Potential angle:

```text
Should signed task receipt IDs live in MCP `_meta`, trace attributes, a companion server, or a client-side lifecycle hook?
```

Draft discussion comment:

```text
I am experimenting with AIPOU, an MCP-first receipt layer for AI agents. It records begin/end task lifecycle metadata as signed receipts while keeping raw prompts and outputs local. I am not proposing this as core protocol yet; I am trying to understand where receipt IDs and validation status should live: tool result `_meta`, trace attributes, a companion MCP server, or client hooks. Any guidance from MCP client/server maintainers would be valuable.
```

Do not post without exact destination confirmation.

Local-agent review:

The `qwen2.5:3b` OpenClaw agent ranked integration surfaces as:

1. Client lifecycle hooks - best fit because receipt creation is a task lifecycle event.
2. Trace attributes - useful for observability, but should avoid leaking sensitive data.
3. Tool result `_meta` - useful for local UI/client state, but not ideal as the main proof transport.
4. Companion MCP server - clean separation, but more integration overhead and possible central point of failure.

Working hypothesis:

```text
AIPOU should integrate first as a client lifecycle hook plus optional trace/span metadata. Tool result `_meta` can expose receipt status to the local client UI, but the signed receipt store should remain local.
```

Local-agent review:

The `qwen2.5:3b` OpenClaw agent was skeptical that scanners need to validate AIPOU directly. The useful interpretation is:

- scanners may not care about AIPOU economics
- scanners may care about contextual provenance fields
- AIPOU should not ask scanners to become validators

Candidate fields for security tooling:

| Field | Why it matters |
| --- | --- |
| `taskId` or `receiptId` | Links a scan finding to a specific agent task. |
| `agentId` or `client` | Identifies which client/agent produced the work. |
| `timestamp` or `recordedAt` | Places the receipt in an audit timeline. |
| `toolOrServerRef` | Links the receipt to the MCP server/tool under review. |
| `validationStatus` | Shows whether the receipt was accepted, rejected, or pending. |

Risks to lead with:

- receipt systems can be spammed by automated agents
- receipt IDs can be misused as trust badges if people do not understand what they prove
- replay protection must be explicit and testable

Better outreach question:

```text
Would receipt/provenance fields help agent-security scanners connect findings to specific MCP tasks, or would that create misleading trust metadata?
```

Local-agent review:

The `qwen2.5:3b` OpenClaw agent ranked framework integration surfaces as:

1. MCP adapter middleware - best first target because it avoids changing core framework logic.
2. Framework lifecycle hook - useful when a framework already exposes task start/end events.
3. External observer - cleanly decoupled, but harder to standardize.
4. Tool wrapper - minimal but too narrow; it can miss non-tool agent work.

Smallest acceptable PR hypothesis:

```text
Add an optional AIPOU/MCP receipt middleware that observes task start/end and emits receipt IDs into run metadata, without requiring token claims or onchain behavior.
```
